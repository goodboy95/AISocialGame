这是一份详细的**平台级页面设计方案**（不包含具体游戏内的交互界面）。

设计核心原则是**“模块化容器”**。我们将网站视为一个游戏启动器（Launcher），游戏本身只是装填在这个容器里的内容模块。这样当你未来想增加“海龟汤”或“剧本杀”时，只需要在后台增加配置，前端会自动渲染出新的入口，而无需重构页面结构。

---

### 总体视觉风格定义 (Design System)
* **风格关键词：** 极简 (Minimalist)、卡片式 (Card-based)、科技感 (Tech-savvy)。
* **配色建议：**
    * **浅色模式：** 灰白背景 (#F5F7FA) + 纯白卡片 (#FFFFFF) + 核心交互色（如电光蓝 #2563EB）。
    * **深色模式：** 深灰背景 (#111827) + 枪灰卡片 (#1F2937) + 霓虹色点缀。
* **排版：** 强调大标题与清晰的层级，减少装饰性线条，用间距（Padding/Margin）来区分区块。

---

### 1. 公共导航栏 (Global Navigation)
* **功能：** 全局导航，状态展示。
* **布局：**
    * **桌面端：** 顶部固定栏 (Sticky Header)。左侧 Logo，中间导航链接（游戏大厅、我的记录、排行榜），右侧用户信息及“充值/会员”入口。
    * **移动端：** 顶部简化栏（Logo + 汉堡菜单/个人头像）。底部采用 Tab Bar（大厅、社区、我的）。
* **扩展性设计：** 导航菜单数据由后端配置，未来增加“社区”或“商城”板块时，直接下发新的 Menu Item 即可。

---

### 2. 首页 / 游戏大厅 (Game Hall & Home)
这是用户进入后的第一站，展示所有可玩的游戏。

* **布局结构：**
    * **Hero 区域：** 一句醒目的 Slogan（如“人类与AI的智力博弈”），配一个“快速开始”按钮（随机匹配）。
    * **游戏卡片网格 (Game Grid)：** 核心区域。采用响应式网格（桌面端一行3-4个，移动端一行1-2个）。
* **卡片设计 (Game Card)：**
    * **封面图：** 游戏的代表性图标或插画（狼人、卧底）。
    * **标题与标签：** 游戏名称 + 标签（如“逻辑推理”、“聚会必备”、“AI高智商”）。
    * **热度/状态：** 当前在线人数或“推荐”角标。
    * **主按钮：** “创建房间”。
* **扩展性设计：**
    * 前端不写死“狼人杀”和“卧底”的入口，而是**遍历后端返回的游戏列表 (GameList Array)** 渲染卡片。
    * 每个游戏对象包含：`game_id`, `name`, `cover_url`, `min_players`, `tags`。未来增加游戏只需在数据库添加一条记录。

---

### 3. 房间列表页 (Room Browser) - 可选
如果不希望用户通过大厅直接创建，而是先看有没有现成的房间。

* **布局：**
    * **顶部筛选器：** 游戏类型（下拉菜单：全部 / 狼人杀 / 卧底）、房间状态（等待中 / 进行中）。
    * **列表区：** 垂直流式布局。
* **单条房间项：**
    * 显示：房主头像、房间名、人数进度（如 5/9）、游戏模式标签。
    * 按钮：如果是“缺人”状态，显示高亮的“加入”；如果是满员，显示灰度“观战”。

---

### 4. 房间创建/配置页 (Create Room / Configuration)
**这是扩展性最关键的页面。** 不要为每个游戏做单独的创建页，而是做一个**动态表单页面**。

* **布局逻辑：**
    * **左侧/顶部（通用设置）：** 所有游戏都需要的设置。
        * 房间名称输入框。
        * 密码/私密性设置。
    * **右侧/底部（游戏特定设置）：** **Dynamic Form 区域**。
* **交互流程：**
    1.  用户在“大厅”点击某个游戏的“创建房间”。
    2.  进入此页面，URL 参数带上 `?gameType=werewolf`。
    3.  页面根据 `gameType` 向后端请求该游戏的配置 Schema（JSON 格式）。
    4.  **前端渲染引擎**根据 Schema 生成表单：
        * 如果是狼人杀，渲染：下拉框（6/9/12人）、开关（屠边/屠城）、多选框（神职配置）。
        * 如果是谁是卧底，渲染：输入框（卧底人数）、下拉框（词库类型）。
* **底部固定栏：**
    * 显示预计消耗的 Token 或 积分。
    * 大大的“创建并进入”按钮。

---

### 5. 等待大厅 (Room Lobby / Waiting Room)
用户创建房间后、游戏正式开始前的状态。这是**真人与AI交互的第一触点**。

* **布局：**
    * **座位网格：** 显示所有座位（空位、真人、AI）。
    * **邀请区：** 分享链接/二维码。
    * **控制台（仅房主可见）：** 位于底部或侧边。
* **核心功能：AI 添加器 (The AI Spawner)**
    * 点击空位，弹出**“添加 AI 玩家”侧边栏/模态框**。
    * **AI 角色市场：** 这是一个列表，展示你后台配置好的 AI 角色。
        * *列表项：* 头像 + 昵称 + 性格标签（如：高冷、话痨、糊涂蛋）。
        * *高级选项：* 允许房主微调该 AI 的 Prompt 参数（如果商业模式允许），例如调整“攻击性 1-100”。
* **扩展性设计：**
    * 座位数量根据游戏类型动态渲染（狼人杀12个坑，卧底6个坑）。
    * AI 列表是从后台拉取的，未来你可以添加新的性格包，无需改代码。

---

### 6. 个人中心 / 资产管理 (User Profile)
用于商业化和用户留存。

* **布局：**
    * **头部：** 用户头像、昵称、UID。
    * **资产卡片：**
        * **余额/Token数：** 显示剩余可用的高级模型调用次数。
        * **充值按钮：** 醒目设计。
    * **数据统计 (Stats)：**
        * 胜率雷达图。
        * 游戏场次历史列表。
    * **设置：** 语言、音效开关、账号绑定。

---

### 7. 后台管理入口 (Admin Dashboard - 简述)
虽然你问的是面向用户的网站，但你的需求里提到了“配置词库”和“AI性格”。建议将这部分做成一个简单的 CMS（内容管理系统）页面，只有管理员账号可见。

* **功能模块：**
    * **游戏配置 (Game Config)：** 管理 Json Schema（决定创建房间时的选项）。
    * **词库管理 (Word Bank)：** CSV 上传或表格编辑。
    * **AI 人设库 (Persona Store)：**
        * 新建角色：输入名称、头像 URL、TTS Voice ID。
        * **System Prompt 编辑器：** 一个大文本框，用于写 Prompt（如“你是一个傲娇的狼人杀玩家...”）。

---

附注：为了实现**“以后增加游戏也能简单加入”**，请在开发时注意：

1.  **不要写死路由 (Hardcode Routes)：** 路由应该是 `/game/:gameId/room/:roomId`，而不是 `/werewolf/room/...`。
2.  **配置驱动 UI (Schema-driven UI)：** 房间设置页必须根据 JSON 渲染，不要手写 `<input>` 标签。
3.  **统一接口格式：** 所有游戏必须遵循统一的 WebSocket 协议标准（例如：`StartGame`, `EndGame`, `PlayerAction`, `SystemBroadcast`），具体游戏逻辑差异由 Payload 内容区分，而不是协议结构区分。
